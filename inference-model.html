Inference Model Training

Case Study: Feature release / My roles: Interaction Design, UI design

Intro

The OpsRamp 5.2 release introduced new features for their AIOps engine that uses machine learning to automatically create incident tickets to swiftly alert the right on-call teams during an outage. Users can train these learning models with their own training data. I worked with the AIOps product team to design and integrate the custom interface and to define the machine learning user experience.

Problem space & research

To have a deeper understanding of the paint points surrounding managing alerts, I first consolidated research from speaking to IT and DevOps experts and users to map out the typical workflow for alert management and which critical points within the workflow that can be optimized by automation.

A quick lesson on terminology:

  • Event: A change to the normal behavior in an environment.
  • Alert: A notification that an event(s) has taken place.
  • Incident: An event that negatively impacts an organization.


In a traditional DevOps alert management workflow, each step requires someone to step in and address it. Despite DevOps adoption models having shown to work well, especially in smaller teams and environments, a rules-based approach doesn't easily scale up at an enterprise level. With changes in infrastructure happening multiple times a day, the average time spent across task flows, per ingested alert, increases disproportionately with the number of incoming alerts. Instead, AIOps allows for automation of these tasks which means alerts donâ€™t have to be manually dealt with.

An AIOps workflow

AIOps takes incoming alerts and groups related alerts into inferences, and then creates a model to make predictions based on these patterns. Users can import their own datasets to bootstrap an inference model. Instead of addressing individual alerts, DevOps teams can manage these inference models.

I worked with the developer and architect to design the end-to-end flow and the interface for the inference models: how users navigate to create and define alert escalation policies, and to train the inference models within those policies.

Design

The bulk of my work with the alert escalation policy creation flow was to design the model training section. In the model training section, users can iteratively change their dataset with data transformation steps until the dataset works well for building a good inference model. The data transformation steps are fundamentally written with the structural conventions of a normal programming language, but the users who tend to manage and create these policies are admin and business users who don't necessarily have a background in data science or programming. So, my goal was to design this data wrangling tool to be accessible for a non-technical crowd.


Design for data literacy

One of the more difficult challenges was to design a way for non-technical users to build their own custom functions from scratch. I worked with the developer to consider all the possible combinations of custom functions a user could create using the functions we would support. Then, we translated code to pseudo-code to conversational English.

Using the custom data transformation fields, the user first creates a condition (if/else if/else statement), and then specifies what modification is applied because of the condition.



Feature rollout

Auto-incident creation and routing, as well as the augmented training for inference models, rolled out during the Winter 2019 release. View the demo on the Webinar introducing all other feature releases.



What I learned

There's rarely a one-size-fits all solution when designing tables for enterprise applications: Though we referred to many guidelines and resources online at the start of designing the table components, our design became increasingly complex even as we descoped our use cases. Ultimately, the best solution was to continually get feedback from the users to alter the design as needed.

Design plays a huge role in making big data more accessible: When the non-technical professional is also equipped with the right tools, they get more comprehensive insights from their data. When we were coming up with the different possible custom functions, we realized that different users had unique ways of writing their own. Designing a way to make this process consistent and easier via a simple interface meant that there was less effort a single user spent on amassing a personal library of custom functions that no one else would be able to use.
